# Roadmap: TiPToP Workspace Robotic Grasping System

## Overview

This roadmap turns an already functioning but locally fragile robotics workspace into a reproducible, debuggable, and regression-safe grasping system. The order starts by stabilizing workspace boundaries and configuration, then hardens the perception and planning contracts, and finally adds verification scaffolding so future work can move faster without rediscovering the same failures.

## Phases

**Phase Numbering:**
- Integer phases (1, 2, 3): Planned milestone work
- Decimal phases (2.1, 2.2): Urgent insertions (marked with INSERTED)

Decimal phases appear between their surrounding integers in numeric order.

- [x] **Phase 3: Perception Chain Stabilization** - Lock in the SAM3 + Fast-FoundationStereo baseline and protect the debug pathways that validate it
- [x] **Phase 4: Planning and Service Contract Hardening** - Preserve reliable grasp/planning behavior across offline, live, and websocket modes
- [x] **Phase 5: Regression and Verification Loop** - Build a durable saved-observation and contract-test safety net for the active stack

## Phase Details

### Phase 1: Workspace Baseline
**Goal**: Make the multi-repo workspace explicit so future work is grounded in the real system, not an incomplete single-repo mental model.
**Depends on**: Nothing (first phase)
**Requirements**: WKSP-01, WKSP-02, WKSP-03
**Success Criteria** (what must be TRUE):
  1. The active workspace repos and their roles are documented from the workspace root.
  2. A new contributor can tell which repos/services are required for the end-to-end grasping stack.
  3. Planning artifacts and developer guidance reflect the true workspace boundary rather than only `tiptop/`.
**Plans**: 3 plans

Plans:
- [x] 01-01: Consolidate workspace-level architecture and dependency documentation
- [x] 01-02: Align project guidance files with the multi-repo workspace boundary
- [x] 01-03: Document repo ownership, service roles, and expected bring-up order

### Phase 2: Config and Bring-Up Hardening
**Goal**: Remove unnecessary hidden workstation assumptions while preserving the currently validated local setup.
**Depends on**: Phase 1
**Requirements**: CONF-01, CONF-02, CONF-03
**Success Criteria** (what must be TRUE):
  1. Critical service paths and runtime locations are configurable without source hunting.
  2. The validated local workstation still works after portability improvements.
  3. Developers have one clear place to inspect or override active stack configuration.
**Plans**: 3 plans

Plans:
- [x] 02-01: Centralize path and service-root resolution for sibling repos
- [x] 02-02: Harden workspace/root-level bring-up and override documentation
- [x] 02-03: Reduce scattered absolute-path assumptions in the active stack

### Phase 3: Perception Chain Stabilization
**Goal**: Treat the current SAM3 + Fast-FoundationStereo perception chain as a first-class baseline and protect the workflows used to validate it.
**Depends on**: Phase 2
**Requirements**: PERC-01, PERC-02, PERC-03, PERC-04
**Success Criteria** (what must be TRUE):
  1. The default TiPToP perception path clearly uses SAM3 and Fast-FoundationStereo.
  2. Focused perception demos remain usable as preflight checks for the full stack.
  3. The SAM3 text-prompt path remains compatible with planner-ready scene generation.
  4. The team can distinguish perception-stage regressions from downstream planning failures quickly.
**Plans**: 4 plans

Plans:
- [x] 03-01: Audit and tighten SAM3 baseline integration points
- [x] 03-02: Audit and tighten Fast-FoundationStereo baseline integration points
- [x] 03-03: Preserve or improve focused debug/demo flows for perception validation
- [x] 03-04: Capture perception-stage expectations and failure signatures explicitly

### Phase 4: Planning and Service Contract Hardening
**Goal**: Keep grasping and planning behavior reliable across the supported execution modes while the multi-service stack evolves.
**Depends on**: Phase 3
**Requirements**: GRSP-01, GRSP-02, GRSP-03, GRSP-04
**Success Criteria** (what must be TRUE):
  1. Offline H5 planning still works with the active perception/planning stack.
  2. M2T2-backed grasping remains functional from the current point-cloud pipeline.
  3. Planner fallback behavior is preserved when provided grasps are missing or unusable.
  4. The websocket planning path stays aligned with simulator expectations.
**Plans**: 4 plans

Plans:
- [x] 04-01: Document and enforce M2T2 contract expectations from TiPToP
- [x] 04-02: Protect offline H5 and serialized-plan execution paths
- [x] 04-03: Lock down planner fallback and degradation behavior
- [x] 04-04: Verify simulator/websocket integration boundaries

### Phase 5: Regression and Verification Loop
**Goal**: Turn today’s manual evidence-heavy validation process into a repeatable regression loop for future development.
**Depends on**: Phase 4
**Requirements**: TEST-01, TEST-02, TEST-03
**Success Criteria** (what must be TRUE):
  1. The main grasping stack has an automated regression path that does not require a live robot.
  2. Saved observations and documented contracts can catch common cross-repo breakages early.
  3. Future changes to SAM3, Fast-FoundationStereo, M2T2, or TiPToP are safer to validate than they are today.
**Plans**: 4 plans

Plans:
- [x] 05-01: Expand saved-observation regression coverage around the active stack
- [x] 05-02: Add explicit cross-service contract tests or smoke checks
- [x] 05-03: Document the expected verification loop for local development
- [x] 05-04: Close the highest-risk test gaps found during earlier phases
