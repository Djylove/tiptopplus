# Requirements: TiPToP Workspace Robotic Grasping System

**Defined:** 2026-04-20
**Core Value:** A real robot can reliably go from camera observations and language/task intent to grasp-ready plans using the current TiPToP + SAM3 + Fast-FoundationStereo + M2T2 stack without fragile workstation-only guesswork.

## v1 Requirements

### Workspace Reproducibility

- [x] **WKSP-01**: A developer can identify every repo and service required for the end-to-end grasping stack from workspace-local project documentation
- [x] **WKSP-02**: A developer can bring up the TiPToP grasping workspace from the workspace root without relying on undocumented sibling-repo assumptions
- [x] **WKSP-03**: Workspace-level planning artifacts explicitly track which sub-repos are part of the active system

### Configuration Portability

- [x] **CONF-01**: The default workspace configuration documents or resolves the active `sam3`, `Fast-FoundationStereo`, `M2T2`, and `droid-sim-evals` dependencies clearly
- [x] **CONF-02**: Critical runtime paths and service locations can be overridden without hand-editing scattered source files
- [x] **CONF-03**: The validated local workstation configuration remains supported while portability improves

### Perception Pipeline Stability

- [x] **PERC-01**: The D435 grasping pipeline uses SAM3 as the default segmentation backend in the main TiPToP flow
- [x] **PERC-02**: The D435 grasping pipeline uses Fast-FoundationStereo as the default FoundationStereo-compatible depth path in the main TiPToP flow
- [x] **PERC-03**: The TiPToP perception pipeline can still produce planner-ready scene outputs when the current SAM3 text-prompt path is enabled
- [x] **PERC-04**: Focused debug/demo entrypoints remain available for validating perception stages before full robot execution

### Grasping and Planning Reliability

- [x] **GRSP-01**: TiPToP can generate M2T2-backed grasp proposals from the active point-cloud pipeline in the current workspace
- [x] **GRSP-02**: TiPToP can produce serialized plans from offline H5 observations using the active pipeline
- [x] **GRSP-03**: Planning fallback behavior for missing or unusable M2T2 grasps is preserved and regression-tested
- [x] **GRSP-04**: The websocket planning server remains compatible with the simulator-side evaluation path

### Verification and Regression Safety

- [x] **TEST-01**: The workspace has documented automated regression coverage for the core `SAM3 + Fast-FoundationStereo + M2T2 + TiPToP` path
- [x] **TEST-02**: The workspace has at least one reproducible saved-observation path that validates the main grasping stack without requiring a live robot
- [x] **TEST-03**: Critical cross-service contracts are documented well enough to detect breaking changes between sibling repos

## v2 Requirements

### Productization and Scale

- **PROD-01**: The workspace supports reproducible setup across multiple operator machines with minimal manual repair
- **PROD-02**: The workspace formalizes version pinning or manifest management across sibling repos
- **PROD-03**: The project adds stronger automated validation for live-hardware execution paths

### Expanded Domain Coverage

- **DOMN-01**: The stabilized grasping stack is generalized beyond the currently validated local scenarios
- **DOMN-02**: Adjacent workflows like fixture cleaning are integrated under a clearer shared system strategy

## Out of Scope

| Feature | Reason |
|---------|--------|
| Cloud-hosted multi-user deployment | Current project is a workstation robotics stack, not a SaaS system |
| Full planner replacement | The immediate goal is to harden the existing TiPToP-based architecture |
| Removing all legacy SAM2 support immediately | Compatibility is still useful while the SAM3 path is being hardened |
| Broad domain generalization before baseline stabilization | The active priority is making the current validated pipeline dependable first |
